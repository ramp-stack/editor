Here's the recap:                                

  ---                                              
  Tree-sitter syntax highlighting — comments not 
  colored                                          
                  
  The traversal loop in build_text_slice iterates  
  leaf nodes and calls token_color to color each   
  one. For line_comment, tree-sitter gives us // as
   a leaf node (kind "//") — the comment body text 
  is not a separate leaf, it lives in the byte
  range of the parent line_comment node.

  The fix needed: When the current leaf kind is    
  "//", look up the parent node, grab its full byte
   range (parent.start_byte()..parent.end_byte()), 
  push that as a single span with theme.comment,
  and advance prev_end to parent.end_byte(). The
  normal gap span
  (source_code[prev_end..start_byte]) should still
  be pushed before it.

  The same investigation is needed for             
  block_comment — we don't yet know if it has the
  same structure.                                  
                  
  string_content is also pending — the match arm in
   token_color needs to be updated from
  "string_literal" to "string_content" since       
  tree-sitter breaks string literals into ",
  string_content, " leaf nodes.

  After both are fixed, remove the debug println!s.
